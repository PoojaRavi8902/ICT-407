18.4 — Data Validation & Data Integrity

Clear, practical explanations for students
Class time: 25–35 min
Concise explanation (teacher wording)

Validation prevents bad data entering the system; integrity keeps the whole database correct, linked and useful.

Validation checks — what they are & quick classroom examples

CheckWhat it doesSimple classroom example
RangeLimits numeric valuesMarks must be 0–100
List / LookupLimits choices to preset valuesGender → Male / Female / Prefer not to say
FormatRequires patternDate: DD/MM/YYYY
PresenceCannot be blankStudent ID required
LengthLimits charactersPostcode = 6 chars
TypeNumeric, text, datePhone number numeric only

Tip for demos: show a CSV with bad rows, import it and use the package's import wizard to highlight rejected records — students see validation in action.

Referential integrity (simple phrasing)

When tables are linked (for example Students and Enrollments), referential integrity makes sure that every enrolment refers to a real student. Tools: Relationships → check Enforce Referential Integrity.

Activity A — class demo (10–12 min)
  1. Open the sample database (Students + Enrolments).
  2. Try to delete a student who has enrolments — observe the package response.
  3. Turn off enforcement (teacher demo only) and delete the student; show the orphaned enrolment.
  4. Discuss consequences and restore from backup or undo.
Assessment: short written note — “What happened and why is enforcement important?” (3 marks)

18.5 — Manipulating Data

Sort, filter, query, insert, update, delete — focus on software steps
Class time: 35–45 min

Sorting & Filtering — student-friendly explanation

Sorting orders records so we can find patterns. Filtering hides records that don’t match a rule. Use multi-field sorts for tidy reports.

Teacher steps — Quick demo (5 steps)
  1. Open the table or query.
  2. Click column header & choose Sort Asc/Desc.
  3. For multi-field sort, open the Sort dialog and add fields in order.
  4. To filter by criteria, use Filter by Form or enter criteria in Query Design.
  5. Demonstrate AND / OR by combining criteria.

Queries — practical view

Explain queries as "questions" the database answers. Show examples: find all stock < 10, list all sales in January, show average salary per dept.

Activity B — pair practical (25–30 min)
  1. Task 1 (10 min): Create a select query that shows students with marks > 75. Save as HighMarks.
  2. Task 2 (10 min): Add a calculated field that converts marks to grade bands (e.g., 90+ → A+).
  3. Task 3 (5 min): Export the query result as CSV and paste a screenshot into the worksheet.
Assessment: submit the query name, screenshot and one-line explanation of the calculated field (5 marks).

Insert / Update / Delete — best-practice reminders

  • Always back up before bulk updates or deletes.
  • Use clear WHERE-like criteria when deleting in queries (example shown in class).
  • Prefer forms for data-entry to reduce accidental mistakes.

18.6 — Reporting from a Database

Turn query output into readable reports
Class time: 25–30 min

Reports should be tailored to the audience: include the fields they need, group related data, and summarise with totals or averages.

  1. Choose source query/table
  2. Design grouping (e.g., department, month)
  3. Add summaries (COUNT, SUM, AVG)
  4. Adjust layout and headings
  5. Preview and export (PDF for distribution)
Activity C — mini report (homework / class)

Create a report from the HighMarks query showing counts per grade band and include a title, page number and date. Export to PDF and submit.

Mark scheme: correct grouping (2), correct totals (2), neat layout (1).

18.7 — Databases in a Wider System Context

Import/export, security, backup and integration
Class time: 20–25 min (discussion)

Import / Export — classroom notes

Common formats: CSV, TXT, XLSX, XML. Explain why CSV is simple & universal (no formatting) while XLSX preserves sheets and formatting.

Security & Backup — teacher points

  • Usernames & passwords, role permissions, audit trails.
  • Encryption is for sensitive data — conceptually explain only.
  • Backups: full and incremental; schedule and test restores.
Activity D — short discussion (10 min)

In groups, students list 3 risks when linking a school database to a public web form. Collect ideas and write one short mitigation for each risk.

Assessment: each group submits 3 risks + mitigations (2 marks each).

Revision checklist & Sample questions

Revision checklist (quick)
  • List validation checks and examples
  • Explain referential integrity with an example
  • Be able to create basic queries, filters and sorts in the software
  • Design a simple grouped report with totals
  • Explain import/export formats and backup strategies
Sample exam-style questions
  1. Define referential integrity and give one example of its importance. (3 marks)
  2. Describe two validation checks and state where you would apply them. (4 marks)
  3. Explain why a manager might prefer a grouped report rather than a raw table. (3 marks)

Lesson plan (2 classes)

  1. Class 1 (75 min): Intro validation + integrity demo (20), sorting/filtering (20), pair practical query (30), quick plenary (5).
  2. Class 2 (75 min): Report design (30), export/import (15), security & backup discussion (15), assessment tasks & submission (15).
