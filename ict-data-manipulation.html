Overview

Chapter 18 covers databases and data manipulation: creating tables, keys and relationships, data entry and editing, sorting/filtering, validation/verification, queries (selection and calculation), forms and reports, importing/exporting, and ethical/legal/security considerations. Each topic below is presented in a lesson-style structure

1. Definition and Uses of a DatabaseStarter • Lecture • Activity

Starter

Ask students: "Where does your school keep student details, marks, attendance?" — use answers to motivate the need for databases.

Lecture

Definition: A database is an organised electronic collection of related data designed for efficient storage, retrieval, updating and reporting.

Purpose & uses: store large related data sets (students, patients, library), reduce duplication, allow multi-user access, and produce reports.

Example: A school database may include tables such as Students, Attendance, and Marks. Using queries, the system can quickly produce a report of students with attendance below 80%.

Activity

  1. List three real-world databases you use (e.g., bank, social media, school).
  2. For each, note the type of data stored and one benefit of digital storage.

Mini-lecture

Compare electronic databases with paper records: speed, search, security and backup advantages.

Review & Exam tips

  • Paper 2: expect definitions and short-answer questions.
  • Paper 3: be ready to identify fields and records in a practical file.
2. Flat-File and Relational DatabasesStructure • pros/cons

Starter

Show a single spreadsheet with repeated teacher names. Ask: "What problem do you see?" (duplication)

Lecture

Flat-file: single-table storage. Simpler but causes duplication, inconsistency and inefficient updates.

Relational: multiple tables linked by keys. Normalises data (reduces duplication) and improves integrity.

Relational design sample:
  • Students(StudentID, Name, DOB, Class)
  • Subjects(SubjectID, SubjectName, TeacherID)
  • Results(ResultID, StudentID, SubjectID, Mark)

Activity

Convert this flat table into at least two related tables (suggestion: separate Students and Results).

Mini-lecture

Introduce normalisation (informal): first idea is to separate repeating groups into their own tables.

Review

Key terms: flat-file, redundancy, relational tables, normalisation.

Lecture

Table = collection of records. Field = column. Record = row. File = the database file (set of related tables).

4. Primary Key and Foreign Key FieldsUniqueness • relationships

Starter

Show two students with identical names. Ask: "How to identify them uniquely?"

Lecture

Primary key: unique identifier for each record (StudentID, BookID). Should not be null and is usually indexed.

Foreign key: a field that references a primary key in another table to create a relationship (e.g., Result.StudentID → Students.StudentID).

[Screenshot: Relationship diagram with Students.StudentID (PK) linked to Results.StudentID (FK)]

Activity

  1. Choose a primary key for a Bookings table.
  2. Identify a suitable foreign key linking Bookings to Customers.

Review

Paper 2/3: expect short answer questions and practical tasks to set PK/FK in design view.

5. Data Types and Field PropertiesField size • masks • validation

Starter

What data type would you use for a phone number? (text, not numeric to preserve leading zeros).

Lecture

Common data types: Text, Number, Date/Time, Currency, Boolean (Yes/No), AutoNumber. Field properties control data entry (Field Size, Input Mask, Default Value, Format).

Example field properties:
  • Phone: Text, Input mask = (000) 000-0000
  • StudentID: Text, Field Size=8
  • DateOfBirth: Date/Time, Format = dd/mm/yyyy

Activity

Set properties for a Book table: ISBN (Text, length 13), Price (Currency), Published (Date).

Review

Paper 3: you may be asked to set field sizes and input masks — practise in Table Design view.

Data can be entered directly into tables (Datasheet view) or via Forms. Use forms with dropdowns, date pickers and validation to reduce errors. Always backup before bulk edits.

Sorting: arrange records by one or more fields (ascending/descending). Filtering: display only records that meet criteria. Combination: sort filtered set.

8. Data Validation and VerificationChecks • rules • methods

Starter

Enter 999 as student age — discuss why this is wrong.

Lecture

Validation: automatic checks (range, format, type, presence, length). Verification: confirm data matches source (double-entry, visual checks).

[Screenshot: Validation rule: Age >=5 AND Age <=20 with custom message]

Activity

  1. Create a validation rule for Marks: 0 <= Marks <= 100 with message "Marks must be between 0 and 100".
  2. Describe how double-entry reduces transcription errors when importing invoices.

Mini-lecture

Validation reduces incorrect entries at source; verification reduces undetected errors after entry.

Review

Paper 3: you may implement validation rules; Paper 2: expect definitions & examples.

9. Queries (Selection and Calculation)SQL • design view • calculated fields

Starter

Question: "How to show all students with marks greater than 80?"

Lecture

Queries extract and compute data. Use selection criteria and arithmetic to build calculated fields. Queries can use logical operators (AND, OR, NOT) and comparison operators (<, >, =, <=, >=).

Access (Design view) example

Field: Name | Marks
Table: Students | Students
Criteria:      | >80

SQL example

-- Select students with marks above 80
SELECT Name, Marks
FROM Students
WHERE Marks > 80
ORDER BY Marks DESC;

Calculated field (SQL)

SELECT StudentID, (Maths + English + ICT) AS TotalMarks
FROM Results;

Activity

  1. Create a query to find students aged < 15.
  2. Create a query that shows Name and TotalMarks (sum of three mark fields).

Mini-lecture

Show how to save queries with meaningful names: qry_Above80, qry_TotalMarks.

Review & Exam tips

  • Paper 3: design and SQL tasks are common; practice both.
  • Paper 2: be able to explain terms such as SELECT, FROM, WHERE.

Forms are interfaces for data entry/viewing. Reports present data for printing or sharing. Reports should include headers, footers, grouping, totals and meet house-style requirements (fonts, logos).

Activity

  1. Create a form for the Students table with navigation buttons.

Mini-lecture

When preparing reports for external audiences, check confidentiality before printing or emailing.

Review

Paper 3: expect tasks that require formatting reports and using grouping/totals.

11. Importing and Exporting DataCSV • TXT • XLS

Starter

How to move a spreadsheet of marks into a database? — discuss format choices.

Lecture

Import: use CSV/TXT/XLS to bring external data in. Export: save query or table to CSV/TXT/XML for sharing. Watch delimiters, date formats and field mapping during import.

[Screenshot: Access Import Text Wizard mapping columns to fields]

Activity

  1. Import a sample CSV file into a table and fix any type mismatches.
  2. Export a query result into CSV and open it in a text editor to inspect delimiters.

Mini-lecture

Always check encoding (UTF-8) to avoid character corruption and preview the first rows before confirming import.

Review

Paper 3: practical tasks often include import/export — practise mapping fields carefully.

Extra Examples & Exercises — Extended Practice

Practical Project — School Database (Extended)

Build a small relational database for a school with these minimal tables:

  • Students(StudentID PK, FirstName, LastName, DOB, Class)
  • Teachers(TeacherID PK, Name, Dept)
  • Subjects(SubjectID PK, SubjectName, TeacherID FK)
  • Results(ResultID PK, StudentID FK, SubjectID FK, Mark)

Tasks (paper 3 style):

  1. Create the tables with appropriate data types and field properties.
  2. Set primary keys and establish relationships.
  3. Create a data-entry form for Results that uses drop-downs for Student and Subject.
  4. Create queries: qry_Above75 (Marks>75), qry_AvgByClass (average mark per class).
  5. Generate a report grouped by Class showing Average Marks and count of students.
  6. Export qry_Above75 as CSV and include a screenshot of the exported file.

Mini SQL Exercises

  1. Write an SQL statement to create a Students table (include StudentID as primary key).
  2. Insert three sample students using INSERT INTO statements.
  3. Write a SELECT query to show names and marks of students with marks > 80, ordered by marks descending.
Suggested SQL answers (click to reveal) 
-- 1. Create table
CREATE TABLE Students (
  StudentID VARCHAR(10) PRIMARY KEY,
  FirstName VARCHAR(50),
  LastName VARCHAR(50),
  DOB DATE,
  Class VARCHAR(10)
);

-- 2. Insert sample rows
INSERT INTO Students VALUES('S001','Aisha','Khan','2009-05-12','10A');
INSERT INTO Students VALUES('S002','Omar','Ali','2009-08-21','10A');
INSERT INTO Students VALUES('S003','Maya','Saleh','2008-11-02','11B');

-- 3. Select top students
SELECT Name, Marks FROM Students
JOIN Results ON Students.StudentID = Results.StudentID
WHERE Marks > 80
ORDER BY Marks DESC;

Model Answers and Teacher Notes

This section contains short model answers and marking guidance for the exam-style questions and practical tasks above. Use it for marking or as teacher notes when delivering the lesson.

Model answers (click to expand)

Example: Define Primary Key (2 marks)

Primary key: a field (or combination) that uniquely identifies each record in a table. (2 marks: 1 for definition, 1 for uniqueness)

Benefits of relational DB (4 marks)

  1. Reduces data duplication (1)
  2. Improves data integrity/consistency (1)
  3. Easier updates and maintenance (1)
  4. Supports complex queries across tables (1)

Validation vs Verification (4 marks)

Validation: automatic computer check at data entry (e.g., range check). Verification: checking data is correct against source (double entry). One example each earns marks.

Import CSV steps (6 marks)

  1. Open Import Wizard and choose CSV file (1)
  2. Choose delimiter (comma) and encoding (UTF-8) (1)
  3. Map CSV columns to target fields (1)
  4. Set correct data types for fields (1)
  5. Preview and check first rows (1)
  6. Fix mismatches after import (1)
